* Overview

=soma-upgrade-prerequisites= automates the analysis and tracking phases of
upgrading elpaca-managed Emacs packages.  It parses the dependency graph,
assesses risks, computes a safe upgrade order, and maintains a JSON progress
tracker that records the status of every init file in the upgrade set throughout the campaign.

The tool provides five subcommands:

- =generate= -- run the full analysis pipeline and produce output files
- =show= -- display the tracker with optional filtering
- =update-status= -- change an entry's status with automatic cascade
- =update-note= -- attach or update a note on an entry
- =validate= -- check consistency between the tracker, data, and git history

* Concepts

The unit of work is the *init file*, not the individual package.  Each init
file (e.g., =soma-magit-init.el=) declares one or more packages.  The tool
collapses package-level dependencies into init-file-level dependencies: if
any package in init file A depends on a package declared by init file B, then
A depends on B.

When a subcommand requires an init file, use its full name (like
=soma-magit-init.el=), not the package name (like =magit=).  The progress
tracker is keyed by init file.

* Prerequisites

- Python 3.12 or later
- [[https://docs.astral.sh/uv/][uv]] (manages the virtual environment and dependencies automatically)

No manual installation of dependencies is needed.  The shell wrapper invokes
=uv run=, which installs everything on first use.

* Inputs

The =generate= subcommand reads these files:

- =~/.emacs.d/soma/inits-upgrades/soma-inits-dependency-graphs.json= (required) --
  the master dependency graph describing all init files and their packages.
- =~/.emacs.d/soma/inits-upgrades/*-upgrade-process.md= -- one per init file
  to be upgraded.  These define the upgrade set; only init files with a
  matching upgrade-process.md are included.
- =~/.emacs.d/soma/inits-upgrades/*-security-review.md= -- one per init file.
  A missing security review causes =generate= to report a FAIL.

The tool also checks =~/.emacs.d/soma/inits/= for init files on disk (used
for orphan package classification).

* Quick Start

Run the analysis pipeline (dry run -- no files are written):

#+begin_example
bin/soma-upgrade-prerequisites generate
#+end_example

Review the report.  When satisfied, write the tracker and derived data:

#+begin_example
bin/soma-upgrade-prerequisites generate --write
#+end_example

View the progress tracker:

#+begin_example
bin/soma-upgrade-prerequisites show
#+end_example

The wrapper script can be invoked from any directory -- it resolves its own
project root automatically.

* Subcommands

** generate

Runs the full analysis pipeline: parses the dependency graph, detects cycles,
identifies the upgrade set, scans for warnings and risks, computes the
topological sort, and assembles a report.

By default this is a dry run -- the report is printed to stdout and no files
are written.  Add =--write= to create or update the progress tracker and
derived data files.

#+begin_example
bin/soma-upgrade-prerequisites generate          # dry run
bin/soma-upgrade-prerequisites generate --write  # write files
#+end_example

When =--write= is used and a tracker already exists, statuses in
=upgraded=, =skipped=, =reverted=, =in-progress=, and =blocked= are
preserved.  (A =blocked= entry whose blockers are no longer in the
dependency chain is automatically reset to =pending=.)  A =.bak= backup
is created before overwriting.

** show

Displays the progress tracker in a human-readable format.

#+begin_example
bin/soma-upgrade-prerequisites show                       # all entries
bin/soma-upgrade-prerequisites show --status pending      # only pending
bin/soma-upgrade-prerequisites show --flags               # only flagged
bin/soma-upgrade-prerequisites show --dependents soma-magit-init.el
#+end_example

The =--dependents= option shows which entries would be affected by a
cascade if the given init file were skipped or reverted: pending and
in-progress entries that would be newly blocked, already-blocked entries
that would gain an additional blocker, plus any other dependents (upgraded,
skipped, etc.) for rollback awareness.

** update-status

Changes an entry's status.  When the new status is =skipped= or =reverted=,
pending and in-progress transitive dependents are set to =blocked=.
Already-blocked dependents gain an additional blocker entry (so they remain
blocked even if one blocker is later removed).

#+begin_example
bin/soma-upgrade-prerequisites update-status soma-magit-init.el in-progress
bin/soma-upgrade-prerequisites update-status soma-magit-init.el upgraded --note "clean upgrade"
bin/soma-upgrade-prerequisites update-status soma-magit-init.el skipped --yes --note "defer to next cycle"
#+end_example

The =--yes= flag skips the cascade confirmation prompt (used when the
decision has already been confirmed in conversation).

The =--force= flag is required when changing a terminal status to a
non-terminal one (e.g., =skipped= back to =pending=).

Reverse cascade runs automatically whenever an entry moves away from
=skipped=, =blocked=, or =reverted= to a status outside that set.  This
unblocks dependents that were blocked solely by this entry.  A blocked entry
with multiple blockers remains blocked until all its blockers are removed.

The =blocked= status cannot be set manually -- it is managed exclusively by
the cascade mechanism.

** update-note

Attaches or replaces a note on an entry without changing its status.

#+begin_example
bin/soma-upgrade-prerequisites update-note soma-magit-init.el --note "waiting on upstream fix"
#+end_example

** validate

Checks that the tracker, derived data, topological sort, and git history are
all consistent.  Git validation runs against the =elpaca-upgrades= branch
(configurable with =--branch=) and expects upgrade commits to match the
pattern =[<init-file>] Upgrade <package> from ...=.  Only the prefix through
=from= is checked by the validator.

#+begin_example
bin/soma-upgrade-prerequisites validate
#+end_example

Runs five checks:

1. *Mapping completeness* -- every package has a corresponding init file
2. *Topological order* -- dependencies appear before dependents
3. *Reverse deps* -- forward and reverse graphs agree
4. *Tracker vs sort* -- tracker entries match the computed sort
5. *Tracker vs git* -- upgraded entries have matching commits; upgrade commits have a matching =upgraded= or =reverted= entry

If all checks pass, prints "All validation checks passed." and exits 0.
Discrepancies are printed grouped by check and the exit code is 1.

* Output Files

By default, output files are written to =~/.emacs.d/soma/inits-upgrades/=
(paths are configurable via command options).  They are
*not* version controlled.

- =upgrade-progress.json= -- the progress tracker (managed by =show=,
  =update-status=, =update-note=, =validate=)
- =derived-dependency-data.json= -- persisted analysis results (loaded
  automatically by the subcommands that need them)

Both files are created by =generate --write=.  A =.bak= backup of the
tracker is created before each overwrite.

* Typical Workflow

1. Run =generate --write= to create the tracker and derived data.
2. Run =show= to review the upgrade order.
3. For each pending entry (in order):
   - Set it to =in-progress= with =update-status=.
   - Perform the upgrade (on a dedicated git branch).
   - Set it to =upgraded= with =update-status=.
4. If an entry must be skipped, run =show --dependents= first to see the
   cascade impact, then =update-status ... skipped=.
5. Run =validate= periodically to check for inconsistencies.
6. If the dependency graph changes, re-run =generate --write= to update.

* Troubleshooting

- *"Derived data not found"* or *"Dependency graph has changed"*: the derived
  data is missing or stale.  Re-run =generate --write= to regenerate it.
- *"Tracker not found"*: run =generate --write= to create the tracker.
- *Validation reports tracker-vs-git discrepancies*: check that upgrade
  commits on the =elpaca-upgrades= branch follow the expected format
  (=[<init-file>] Upgrade ...=).  See =validate --help= for options.
